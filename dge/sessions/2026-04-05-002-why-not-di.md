# DGE Session: なぜ DI ではないのか

**Date:** 2026-04-05
**Decision:** [DD-001](../../docs/design-decisions.md#dd-001-why-not-di)
**Cast:** ☕ ヤン + 👤 今泉 + ⚔ リヴァイ + 🏥 ハウス + 🎩 千石
**Context:** PropStack が DI フレームワークではなく Registry パターンを選んだ理由。Service Locator は本当にアンチパターンか？作者は「Service Locator が悪いって言ってる人は流されてるだけで本質を理解していない」という強い意見を持っている。

---

🎩 **千石**: まず事実を整理しましょう。Martin Fowler が 2004 年に「Inversion of Control Containers and the Dependency Injection pattern」を書いた。ここで Service Locator と DI を **対等な選択肢** として紹介している。「DI の方が優れている」とは **書いていない**。

👤 **今泉**: そもそも、Service Locator が「アンチパターン」になったのっていつからですか？

🎩 **千石**: Mark Seemann が 2010 年に「Service Locator is an Anti-Pattern」というブログを書いてからです。理由は3つ:
1. API が嘘をつく（コンストラクタに依存が見えない）
2. テストが困難
3. 実行時エラーになる（コンパイル時に検出できない）

☕ **ヤン**: でもさ、それ **Spring の `@Autowired` も全く同じ問題** じゃない？

🏥 **ハウス**: そう。全員嘘をついている。Spring の `@Autowired` は:
- フィールドインジェクションは **コンストラクタに依存が見えない** → Seemann の批判そのまま
- `@Qualifier` がないと実行時に `NoUniqueBeanDefinitionException` → 実行時エラー
- CGLIB プロキシで **デバッガが追えないコールスタック** を生む
- `@Conditional` で **同じコードが環境によって違う振る舞い** をする

つまり Spring DI は **Service Locator の問題を全部持った上で、さらにプロキシの黒魔術が乗る**。

⚔ **リヴァイ**: Guice の方がマシだ。なぜか。

☕ **ヤン**: Guice は **明示的バインディング**:
```java
// Guice — 何が何に紐づくか全部見える
bind(DataSource.class).annotatedWith(Names.named("prod")).to(ProdDataSource.class);
```

Spring は **暗黙のスキャン**:
```java
// Spring — クラスパスを全部スキャンして「見つかったやつ」を注入
@ComponentScan("com.example") // 何が登録されるかはスキャンしないと分からない
```

🎩 **千石**: Guice は「DIのメリットを、Service Locator の明示性で実現した」設計です。

👤 **今泉**: じゃあ「誰が困るの？」を具体的に。Service Locator で困るケースは？

🏥 **ハウス**: 本当に困るケースを診断しよう:

| 症状 | Service Locator | Spring DI | PropStack Registry |
|------|----------------|-----------|-------------------|
| 依存が見えない | `Locator.get(X.class)` がメソッド中に隠れる | `@Autowired` がフィールドに隠れる | 同じ |
| テスト困難 | `put()` で差し替え可 | `@MockBean` で差し替え可 | `put()` で差し替え可 |
| 実行時エラー | 未登録で NPE/例外 | Bean 未定義で起動失敗 | 未登録で例外 |
| デバッグ | **呼び出しが直接的** | **Proxy で追えない** | **呼び出しが直接的** |
| 循環依存 | コンパイルエラーにならない | Spring も検出しない (field injection) | 同じ |

⚔ **リヴァイ**: 要するに **差はほぼない**。だが Spring DI は Proxy + AOP + 暗黙スキャンという追加の複雑さを持つ。

☕ **ヤン**: そしてここが核心。**DI の本当の価値は「依存の注入」であって「フレームワーク」ではない**:
```java
// これが DI の本質。フレームワーク不要。
class MyService {
    private final DataSource ds;
    MyService(DataSource ds) { this.ds = ds; }  // コンストラクタインジェクション
}

// 誰が注入するかは自由
new MyService(registry.get(DataSource.class));  // Registry でも
new MyService(prodDs);                          // 手動でも
new MyService(mockDs);                          // テストでも
```

👤 **今泉**: 「要するに」、DI パターンは良い。DI フレームワークが問題を持ち込む。Service Locator は DI パターンと **組み合わせて使える**。対立軸じゃない。

🏥 **ハウス**: 診断結果。**業界の「Service Locator はアンチパターン」は誤診**。正確な診断は:

> **「グローバルな可変状態で依存を隠すのがアンチパターン」**
>
> Service Locator がそうなりがちだから叩かれたが、Spring の `@Autowired` フィールドインジェクションも **同じ病気**。
> コンストラクタインジェクション + 必要な場所で Registry を使うのは **健全な設計**。

🎩 **千石**: propstack の `Registry` が Spring DI より優れている点を明確にしましょう:

1. **0 依存** — Spring は 20+ の依存を引き込む
2. **起動 0ms** — Spring は ComponentScan + Proxy 生成で数秒〜30秒
3. **デバッグ透明** — Proxy がないからスタックトレースが綺麗
4. **明示的** — `Registry.put()` / `Registry.get()` で何が登録されてるか **読めばわかる**
5. **テスト容易** — `Registry.clear()` + `Registry.put()` でモック差し替え

⚔ **リヴァイ**: 結論出せ。

☕ **ヤン**: こう:

> **DI の原則は正しい。DI フレームワークは過剰。**
> **Service Locator は DI の競合じゃなく補完。**
> **propstack の Registry は「DI の原則 + Service Locator の明示性」を 0 依存で実現する。**

---

**作者のコメント:** まじ、Service Locator が悪くって言っている人は流されているだけで自分の頭で考えてないんじゃないかって思うよ。本質を理解していないｗ

**→ Registry パターン採用。DI フレームワーク不採用。**
